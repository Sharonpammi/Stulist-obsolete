Website Description:

This website is a place where college students can sell/trade various items (books, computers, clothing, etc) or services(tutoring, dance lessons, music lessons, etc.) with other college students. Proof of college status is required in order to make an account (.edu email). However, the search and browse feature isn’t only for users. Anyone will be able to search or browse the website; however, certain information will not be displayed unless logged into an account. User’s cannot directly buy items through the website, they have to collaborate with the seller and decide on the final price/meeting place for the transaction. Buyers and sellers will have a medium in which they can communicate with each other.

Account Creation:

  • Users can create an account by providing an .edu email address, full name, university name, account username, a password, and travel range. Necessary validation will be done (Ex: check if username exists, validate email, etc..)
  • Once created and confirmed, the user can list items up for sale/trade as well as search/browse for items with the ability to see seller/price information. (Non account holders can’t see seller info or prices).

Users can list items/services for sale/trade:

  • An account is necessary for this feature of the website.
  • Once logged in, the user can click on a button that will guide him or her to the listing webpage.
  • In order for the item/service to be listed, all necessary filters and specific information must be provided. User’s can’t omit certain filters because they will be needed when other users search for items.
  • Use Case:
    • User wants to sell a History textbook.
    • After arriving to the listing webpage, he or she has to select the category in which the item belongs to (In our case: Textbooks).
    • Based on the category, users will be prompted to provide additional sub-categories. (In our case, the sub category will be subjects: History)
    • User category selection is completed, the user will be prompted with text boxes for various item descriptions (Title, Author, ISBN, ISBN 13)
    • The user can then provide a brief description of the item, and must select the condition of the item (New, Like New, Very Good, Good, Damaged, Broken, etc.)
    • Finally, the user can choose whether to list the item as sell and/or trade. If “sell” is selected, the user must input the price at which the item will be sold for. If “trade” is selected, the user can specify the price range of items that he or she is willing to trade for.

Users can search for certain items/services:

  • The main search bar will only execute the user’s search once a category is selected.
    • The categories will be provided next to the search bar in the form of a drop down menu.
    • The user has to specify one of the two major categories (items or services)
    • Depending on which major category is selected, sub categories will be provided as additional filters.
    • This is also where the user can specify the rating limit on items, based on the ratings. (Ex: only list items that are at or above 4 stars).
    • Once a category is provided, depending on which is selected, additional text boxes/GUI properties (check boxes, radio buttons, etc…) will appear for filtering appropriate results.
  • Use Case:
    • Let’s say a user wishes to search for textbooks. He or she will select the items category, and select Textbooks.
    • Once the category is selected, another drop down menu will appear, allowing the user to specify the topic of their search (Title, Author, ISBN, ISBN 13, etc…)
    • Once the search topic is selected, another drop down menu will appear, allowing the user to specify the subject of interest (Math, History, Physics, Computer Science, etc…). It’s important to note, this portion is optional. If the user wanted, he can press the search button and view results without providing additional filters.
    • Once submitted, the user will be provided with a list of all items that are in relevance to his or her search.
  • The search feature is not limited to users who have accounts. Anyone will be able to search items. However, the price, individual item description page, and seller information will require an account to be viewed.

Users can browsed by item/service categories:

  • Users can select major categories to browse by (Computers, Textbooks, Clothing, Tutoring, etc...)
  • Just like the search filter, user’s can select rating limits on items/services (Ex: Higher than 3.5 stars).
  • Users will then be presented with a list of all items within that result.
  • If needed, the user can refine these results by providing additional filters (items that range between certain prices, items that are only up for trade, Tshirts *if in the Clothing category*, etc...)
  • Users can then click on any item/service and will be directed to the item/service description page.
  • The browse feature is not limited to users who have accounts. Anyone will be able to search items. However, the price, individual item description page, and seller information will require an account to be viewed.

Rating System:

  • Based on the quality of a transaction, both buyers and sellers can rate each other.
  • The review is based on five-star scale (higher is better).
  • Users can select a rating in increments of .5 (0.5 being the lowest and 5.0 being the highest)
  • A seller’s/buyer’s rating is calculated with the percentage of overall reviews. For example, a user has 5 reviews in total. The score of each review is 3, 4.5, 3, 4, 2.5. The total score is 17. The percentage is 17/25 = 68%
  • This information will be used as additional filtering options for when users search/browse for items.

User-to-user communication:

  • Once a buyer is interested in an item or service, he or she has to first be logged into an account. The buyer can then send the seller a notification with a brief message.
  • The notification will appear in the inbox of the seller.
  • If the seller wishes to sell the item to the buyer, he or she can send a message back, which will in turn appear in the buyer’s inbox.
  • By default, the message will appear in the user’s account inbox. However, if the user gives permission to allow email notifications, the message will also be emailed to the user’s .edu email. (However, the user has to sign into the website to send a response. Replying to the email won’t be sent to the buyer/seller.)
  • The seller and buyers will discuss and make the necessary meeting arrangements/final price.
  • Once an item is sold/traded, the seller can remove his or item from being listed.
    • Listed items must be given a time frame as to how long they are to stay active. After the given time, the listing will be automatically taken down regardless if the item hasn’t been sold/traded
    • The maximum length of time listings may stay active is 14 days. If a seller posts a listing to stay active for 14 days, sells the item earlier, he can either close the listing himself or let our system take down the listing after 14 days has past.

Google API Integration:

  • One of the services that StuList has to offer is allowing our users to define how far (in miles) they are willing to travel to make a transaction occur. The proximity is based off the college a student attends. (For example, a student at CSULB who wants to sell a book on StuList can list that he/or she is willing to travel in a 15 mile radius from CSULB.)
  • Google Maps API gives us the tools to calculate the range of how far our users would be willing to travel to sell and buy from others on StuList.
  • Google Maps API is a powerful tool as it gives our site navigational services for our users.
  • It is important to note that this API implementation is one of our lowest priorities as of this moment. Once the main parts of our website is working properly, we will then try our best to research/implement this feature into our website.
  • Furthermore, as Google Maps API integration is just a side project compared to the overall premise of creating StuList, we can only disclose conceptual ideas and services we’d be able to provide with Google Maps API.
